# Art Pipeline Innovation — Claude Code First

Ideas for making Crusty a joy to create visual art in, with Claude as creative partner.

> **Philosophy**: Every idea below assumes a developer sitting in their terminal with Claude Code open, working in Rust. No external editors required. The engine's software framebuffer (raw RGBA pixels) is the canvas. Claude reads pixels, writes pixels, and iterates with you.

---

## 1. Sprite Sheet Importer (`crusty import-spritesheet`)

**The problem**: You bought a tileset from itch.io or hired an artist. You have a PNG. Getting it into the engine means manually writing raw RGBA byte arrays — painful and error-prone.

**The idea**: A CLI command that takes a PNG file, tile dimensions, and optional metadata, and generates a Rust module containing the sprite sheet data as embedded bytes, tile index constants, and animation clip definitions.

```bash
crusty import-spritesheet assets/dungeon_tiles.png --tile-size 16x16 --name dungeon
```

Generates:
```rust
// Generated by `crusty import-spritesheet`
pub const SHEET_DATA: &[u8] = include_bytes!("../assets/dungeon_tiles.rgba");
pub const SHEET_WIDTH: u32 = 256;
pub const SHEET_HEIGHT: u32 = 256;
pub const TILE_W: u32 = 16;
pub const TILE_H: u32 = 16;

// Named tile indices (auto-detected or user-annotated)
pub const FLOOR_STONE: u32 = 0;
pub const WALL_TOP: u32 = 1;
// ...
```

**Claude Code workflow**: "Hey Claude, import this sprite sheet and set up walk animations using tiles 0-7 at 0.1s per frame." Claude runs the CLI, reads the output, wires up the `SpriteAnimator` clips, done.

**Implementation**: Add a `png` crate dependency to `engine-cli` only (not engine-core, preserving the boundary). CLI decodes PNG to raw RGBA, writes `.rgba` binary + a `.rs` constants file. Optionally writes a `.json` manifest for the web host layer.

## 2. Procedural Pixel Art Generator (`PixelArtGen`)

**The problem**: Placeholder art is ugly rectangles. Even for prototyping, you want something that *looks like a game*.

**The idea**: A deterministic, seed-based pixel art generator built into engine-core. Given a seed and a style enum, it produces a `SpriteSheet` of coherent tiles — characters, terrain, items.

```rust
pub enum PixelArtStyle {
    Character { width: u32, height: u32, symmetry: bool },
    Terrain { variant: TerrainType },
    Item { silhouette: ItemShape },
    Icon { glyph: char, padding: u32 },
}

pub fn generate_sprite(seed: u64, style: PixelArtStyle, palette: &ColorPalette) -> SpriteSheet
```

**How it works**:
- `Character`: Generates bilaterally symmetric pixel blobs (like the classic spaceship generator). Uses the palette's primary/secondary/accent for body/detail/highlight.
- `Terrain`: Noise-based tile patterns (dirt, grass, stone, water) with auto-tiling edge variants. Uses the palette to tint everything cohesively.
- `Item`: Simple silhouette fills (sword shape, potion shape, key shape) with palette-driven coloring.
- `Icon`: Renders a character from the existing 5x7 bitmap font at larger scale with anti-aliasing and palette colors.

**Claude Code workflow**: "Generate a character sprite sheet with 8 walk frames, using the neon_cave palette." Claude writes the `generate_sprite` call, the engine produces deterministic art from the game's seed. You iterate by adjusting palette, seed, or parameters — all in code.

**Key insight**: Since the engine already has `ColorPalette::from_seed()` and `SeededRng`, every generated sprite is deterministic. Same seed = same art. Different seed = visual variety. This composes with the existing headless sweep infrastructure — run 1000 seeds, screenshot each, pick the one that looks best.

## 3. Claude-Driven Pixel Art Conversation Loop

**The problem**: You want custom art but can't draw. You want to describe what you want and iterate.

**The idea**: A workflow where Claude generates pixel art as Rust code (raw RGBA arrays or framebuffer draw calls), renders it to the software framebuffer, exports a screenshot, reads it back, and iterates.

**The loop**:
1. You describe: *"I want a 16x16 sword sprite, silver blade, gold hilt, red gem in the pommel"*
2. Claude writes a function that uses `Framebuffer` drawing primitives (`fill_rect`, `fill_circle`, `set_pixel`) to paint the sprite pixel by pixel
3. Claude runs a headless test that renders to framebuffer and exports the pixels as a `.ppm` or `.bmp` file (trivial format, no dependencies)
4. Claude reads the exported image to visually verify it
5. You say *"make the blade thinner"* — Claude adjusts the draw calls, re-renders, re-reads

**Implementation**: Add a `headless::export_framebuffer_ppm(fb: &Framebuffer, path: &str)` function. PPM is a dead-simple image format (header + raw RGB bytes). No external crates needed. Claude can read the file back to verify.

```rust
// In headless module
pub fn export_framebuffer_ppm(fb: &Framebuffer, path: &str) -> std::io::Result<()> {
    let mut f = std::fs::File::create(path)?;
    write!(f, "P6\n{} {}\n255\n", fb.width, fb.height)?;
    for i in (0..fb.pixels.len()).step_by(4) {
        f.write_all(&fb.pixels[i..i+3])?; // RGB, skip A
    }
    Ok(())
}
```

**CLI command**: `crusty render-sprite --code sprites/sword.rs --output sword.ppm`

## 4. Tile Map Painter (Code-First)

**The problem**: Level design in code means typing tile indices by hand. Visual tilemap editors exist but break the Claude Code-first workflow.

**The idea**: A declarative ASCII-art-to-tilemap converter. You design levels as ASCII art in your Rust source (or a `.map` text file), and a function converts it to a `TileMap`.

```rust
let map = TileMap::from_ascii(r#"
    ################
    #..............#
    #..##....##....#
    #..##....##..P.#
    #..............#
    #....WWWW......#
    #..............#
    ################
"#, &legend);
```

Where `legend` maps characters to tile configurations:

```rust
let mut legend = AsciiLegend::new(16.0); // tile_size
legend.map('#', TileType::Solid, palette.get(PaletteRole::Primary), Some(WALL_SPRITE));
legend.map('.', TileType::Empty, Color::TRANSPARENT, Some(FLOOR_SPRITE));
legend.map('W', TileType::Solid, palette.get(PaletteRole::Secondary), Some(WATER_SPRITE));
legend.map('P', TileType::Empty, Color::TRANSPARENT, None); // Player spawn marker
```

**Claude Code workflow**: "Claude, design a dungeon level with a treasure room in the top right." Claude writes ASCII art, maps it to tiles, and you can see the layout right in your source code. Iterate by editing the ASCII. Claude can also *read* an existing ASCII map and suggest improvements.

**Bonus: Bidirectional conversion**. `TileMap::to_ascii(&legend)` exports a map back to ASCII art so Claude can read and reason about level geometry even in headless mode.

## 5. Palette-Driven Art Coherence System

**The problem**: When mixing purchased art, generated art, and hand-drawn art, visual coherence falls apart. Different color temperatures, saturation levels, and contrast ratios clash.

**The idea**: Extend `ColorPalette` into a full art direction system that can re-palette existing sprite sheets.

```rust
impl SpriteSheet {
    /// Remap all pixel colors to the nearest match in the given palette.
    /// Uses perceptual color distance (CIE76 or simple weighted Euclidean).
    pub fn remap_to_palette(&mut self, palette: &[Color]) { ... }

    /// Tint the entire sheet toward a target color by a given strength (0.0-1.0).
    pub fn tint(&mut self, color: Color, strength: f64) { ... }

    /// Adjust brightness and contrast.
    pub fn adjust_levels(&mut self, brightness: f64, contrast: f64) { ... }

    /// Reduce to N colors using median-cut quantization.
    pub fn quantize(&mut self, max_colors: usize) { ... }

    /// Extract the unique colors used in this sheet (ignoring alpha=0).
    pub fn extract_palette(&self) -> Vec<Color> { ... }
}
```

**Claude Code workflow**: "I bought this forest tileset but it clashes with my neon_cave palette. Claude, remap it." Claude calls `sheet.remap_to_palette(&neon_cave_colors)`, renders a preview, you iterate on the mapping.

**Implementation**: All operations work on the raw `Vec<u8>` pixel buffer. No external dependencies. Color distance can use a simple weighted Euclidean: `sqrt(2*dr^2 + 4*dg^2 + 3*db^2)` which approximates perceptual distance.

## 6. Nine-Slice Sprite System

**The problem**: UI panels, dialogue boxes, and variable-size decorations need to scale without stretching. Currently you'd have to manually compute and draw 9 separate blit calls.

**The idea**: A `NineSlice` struct that wraps a sprite region and knows how to render at any size.

```rust
pub struct NineSlice {

    pub border_right: u32,
    pub border_bottom: u32,
    pub border_left: u32,
}

impl NineSlice {
    pub fn draw(&self, fb: &mut Framebuffer, sheet: &SpriteSheet,
                x: i32, y: i32, width: u32, height: u32) { ... }
}
```

**Claude Code workflow**: "Claude, make a dialogue box using the stone border from my tileset. It should be 200x80 pixels." Claude wires up the nine-slice with the right tile indices and border insets.

## 7. Sprite Composition & Layering System

**The problem**: Characters with equipment, enemies with damage states, terrain with decorations — all need sprite layering. Currently each entity gets one `Visual::Sprite`.

**The idea**: A `CompositeSprite` component that renders multiple sprite layers per entity with per-layer offsets, tinting, and animation.

```rust
pub struct SpriteLayer {
    pub sheet_id: u32,
    pub tile_index: u32,
    pub offset_x: i32,
    pub offset_y: i32,
    pub tint: Option<Color>,
    pub flip_x: bool,
    pub flip_y: bool,
    pub visible: bool,
}

pub struct CompositeSprite {
    pub layers: Vec<SpriteLayer>,
}
```

**Example**: A character with base body (layer 0), armor overlay (layer 1), weapon (layer 2), and damage flash (layer 3). Change equipment by swapping sheet_id/tile_index on layers 1-2. Flash damage by toggling layer 3 visibility.

**Claude Code workflow**: "Claude, add a helmet layer to the player sprite that changes when they pick up the iron helmet item." Claude adds a `SpriteLayer` to the composite, wires it to the inventory system.

## 8. Animation State Machine

**The problem**: `SpriteAnimator` handles clip playback but doesn't manage transitions. Games need "when walk clip ends and velocity is 0, transition to idle" logic. This currently lives in ad-hoc `step()` code.

**The idea**: A declarative animation state machine that lives alongside `SpriteAnimator`.

```rust
pub struct AnimTransition {
    pub from: String,         // clip name (or "*" for any)
    pub to: String,           // target clip name
    pub condition: AnimCondition,
}

pub enum AnimCondition {
    ClipFinished,                           // Current clip completed (non-looping)
    StateEquals(String, StateValue),        // game_state key check
    VelocityBelow(f64),                     // |velocity| < threshold
    VelocityAbove(f64),                     // |velocity| > threshold
    OnGround(bool),                         // grounded state
    Custom(String),                         // named flag checked by game code
}

pub struct AnimStateMachine {
    pub transitions: Vec<AnimTransition>,
}
```

**Claude Code workflow**: "Claude, set up animations so the player goes idle when standing still, walks when moving, and plays the attack animation when I press space." Claude defines the state machine declaratively, no manual `if/else` chains in `step()`.

## 9. Auto-Tiling System

**The problem**: When you change a tile in a tilemap, adjacent tiles need to update their sprite to show correct wall edges, corners, and transitions. This is the #1 tedious part of working with purchased tilesets.

**The idea**: An auto-tile resolver built into `TileMap` that uses bitmasking to select the correct sprite variant.

```rust
pub struct AutoTileRules {
    /// Maps a 4-bit (cardinal) or 8-bit (cardinal+diagonal) bitmask to a sprite index.
    pub bitmask_to_sprite: HashMap<u8, u16>,
    /// Which TileTypes count as "same" for adjacency checks.
    pub same_types: Vec<TileType>,
}

impl TileMap {
    /// Recalculate sprite_index for all tiles in a layer based on auto-tile rules.
    pub fn apply_auto_tile(&mut self, layer: usize, rules: &AutoTileRules) { ... }

    /// Recalculate only tiles adjacent to (x, y) — for incremental updates.
    pub fn update_auto_tile_around(&mut self, layer: usize, x: usize, y: usize,
                                    rules: &AutoTileRules) { ... }
}
```

**Standard bitmask layouts**: Support the common 47-tile "blob" tileset format and the simpler 16-tile cardinal format that most purchased tilesets use.

**Claude Code workflow**: "Claude, I bought this RPG tileset that uses the standard 47-tile blob layout. Set up auto-tiling for walls." Claude maps the tileset's sprite indices to the bitmask table, applies it. You place walls as `TileType::Solid` and the sprites resolve automatically.

## 11. Sprite Sheet Viewer & Inspector (CLI + Web)

**The problem**: You loaded a sprite sheet but can't easily see what tile index maps to what visual.

**The idea**: A CLI command that renders a sprite sheet as a labeled grid image, numbering each tile.

```bash
crusty inspect-spritesheet --sheet-data assets/dungeon.rgba --width 256 --height 256 --tile 16x16 --output grid.ppm
```

Renders a grid image with tile index numbers overlaid on each tile. Uses the existing `text::draw_text` for labels.

**Claude Code workflow**: Claude runs the inspector, reads the output image, and can reference tiles by index: "Tile 14 is the chest, tile 15 is the chest-open variant."

## 12. Color Palette Preview Renderer

**The problem**: `ColorPalette::from_seed(42)` produces a palette, but you can't see it without running the full game.

**The idea**: A CLI command (or headless utility) that renders palette swatches to an image.

```bash
crusty palette-preview --seed 42 --output palette.ppm
crusty palette-preview --builtin neon_cave --output neon.ppm
crusty palette-preview --hue 210 --scheme triadic --output triadic.ppm
```

Renders a swatch strip showing all 7 roles with labels. Can also render multiple palettes side by side for comparison.

**Claude Code workflow**: "Claude, show me 5 palette variations for seed range 100-104." Claude runs 5 preview commands, reads all images, recommends the best fit for the game's mood.

## 14. Sprite Outline & Shadow Effects

**The problem**: Sprites need outlines for readability against busy backgrounds, and drop shadows for depth. Currently you'd have to manually draw expanded copies.

**The idea**: Rendering utilities that work on any `SpriteSheet` tile.

```rust
impl SpriteSheet {
    /// Draw a tile with a 1px outline in the given color.
    /// Renders the outline first (expanded), then the sprite on top.
    pub fn draw_tile_outlined(
        &self, fb: &mut Framebuffer, index: u32,
        dx: i32, dy: i32, outline_color: Color,
    ) { ... }

    /// Draw a tile with a drop shadow offset.
    pub fn draw_tile_shadowed(
        &self, fb: &mut Framebuffer, index: u32,
        dx: i32, dy: i32, shadow_offset_x: i32, shadow_offset_y: i32,
        shadow_color: Color,
    ) { ... }
}
```

**Implementation**: The outline version draws the sprite 8 times (one pixel in each direction) using the outline color, then draws the real sprite on top. Simple, effective, no shader needed.

```rust
pub struct RoomTemplateLibrary {

impl RoomTemplateLibrary {
    /// Pick a random room matching the given tags.

    pub fn stamp(&self, map: &mut TileMap, template: &RoomTemplate,
```

**Claude Code workflow**: "Claude, create 5 room templates for the dungeon — a treasure room, a boss arena, two corridors, and a puzzle room." Claude writes ASCII art for each, tags them, adds entry points. The procedural generator uses them automatically.

## Design Principles

1. **No external dependencies in engine-core**. Image decoding happens in engine-cli or the host layer. Engine-core only sees raw RGBA bytes.
2. **Deterministic everything**. All generated art uses `SeededRng`. Same seed = same sprites.
3. **Claude can see what it creates**. Every visual tool has a headless export path (PPM/BMP) that Claude can read back to verify.
4. **Composable with existing systems**. Every idea plugs into `SpriteSheet`, `TileMap`, `ColorPalette`, and `SpriteAnimator` — no parallel rendering stacks.
5. **Code is the source of truth**. No external editor state files. ASCII maps, palette seeds, and sprite generation params all live in `.rs` files under version control.
