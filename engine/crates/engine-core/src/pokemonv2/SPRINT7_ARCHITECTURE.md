# Sprint 7 Architecture: Complete Move Database (All 251 Moves)

## Situation

Sprint 6 delivered all 251 species in `species_data.rs` using a const-array pattern auto-generated from pokecrystal ASM. Move data is still scattered across `data.rs` as 21 manually-defined `static MoveData` entries with a `match`-based `move_data()` dispatcher. Unknown move IDs fall back to Tackle.

**Problem:** The `MoveData` struct is missing fields needed for battle mechanics (effect, effect chance, priority, high crit). And 230 of 251 moves have no data at all — they silently return Tackle stats.

**Solution:** Follow the Sprint 6 pattern exactly. Create `move_data.rs` with all 251 moves in a const array, auto-generated from `pokecrystal-master/data/moves/moves.asm`.

## Canonical Data Source

**Single file for all 251 moves:**
`crates/engine-core/src/pokemon/pokecrystal-master/data/moves/moves.asm`

Each entry is one line in fixed format:
```asm
move NAME, EFFECT_*, power, TYPE, accuracy%, pp, effect_chance%
```

Example entries:
```asm
move POUND,        EFFECT_NORMAL_HIT,         40, NORMAL,       100, 35,   0
move FIRE_PUNCH,   EFFECT_BURN_HIT,           75, FIRE,         100, 15,  10
move CURSE,        EFFECT_CURSE,               0, CURSE_TYPE,   100, 10,   0
```

**Supporting data files (same directory):**

- `effects_priorities.asm` — 6 effects with non-default priority:
  ```
  EFFECT_PROTECT      -> priority 3
  EFFECT_ENDURE       -> priority 3
  EFFECT_PRIORITY_HIT -> priority 2
  EFFECT_FORCE_SWITCH -> priority 0
  EFFECT_COUNTER      -> priority 0
  EFFECT_MIRROR_COAT  -> priority 0
  All others           -> priority 1 (default)
  ```

- `critical_hit_moves.asm` — 7 moves with high crit rate:
  ```
  KARATE_CHOP, RAZOR_WIND, RAZOR_LEAF, CRABHAMMER, SLASH, AEROBLAST, CROSS_CHOP
  ```

## Updated MoveData Struct

```rust
#[derive(Clone, Debug)]
pub struct MoveData {
    pub id: MoveId,
    pub name: &'static str,
    pub move_type: PokemonType,
    pub power: u8,
    pub accuracy: u8,
    pub pp: u8,
    pub is_special: bool,
    // New fields (Sprint 7):
    pub effect_id: u8,      // numeric ID for the move effect
    pub effect_chance: u8,   // 0-100, secondary effect proc rate
    pub priority: i8,        // -1 to +3, default 1
    pub high_crit: bool,     // true for the 7 high-crit moves
}
```

**Why `effect_id: u8` instead of an enum:**
- The ASM defines ~90 distinct effect names (EFFECT_NORMAL_HIT, EFFECT_BURN_HIT, EFFECT_MULTI_HIT, etc.)
- Creating a 90-variant Rust enum adds complexity for zero current benefit — we don't implement move effects yet
- A `u8` is compact, can be matched on when we do implement effects, and maps directly to the ASM ordering
- We define `pub const EFFECT_*: u8` constants for each effect, so code reads like `if md.effect_id == EFFECT_BURN_HIT { ... }`
- If/when we implement a full effects system (future sprint), we can convert to an enum at that point

**`is_special` derivation (Gen 2 physical/special split by type):**
- Physical: Normal, Fighting, Poison, Ground, Flying, Bug, Rock, Ghost, Steel
- Special: Fire, Water, Electric, Grass, Ice, Psychic, Dragon, Dark

**`CURSE_TYPE` special case:**
- In the ASM, Curse uses `CURSE_TYPE` (a pseudo-type unique to Gen 2)
- We map it to `PokemonType::Ghost` since Curse's behavior depends on whether the user is Ghost-type, not on the move's type for damage calculation
- Curse is a status move (power=0), so the type only matters for display, not damage

## File Structure

### New: `move_data.rs`

```rust
// AI-INSTRUCTIONS: pokemonv2/move_data.rs — All 251 move data from pokecrystal-master.
// Generated by /tmp/gen_moves.py from pokecrystal ASM source.
// Do not manually edit — regenerate with the script.
use super::data::{MoveData, MoveId, PokemonType};

pub const NUM_MOVES: usize = 251;

// === MoveId Constants (1-251, matching ASM order) ===
pub const MOVE_POUND: MoveId = 1;
pub const MOVE_KARATE_CHOP: MoveId = 2;
// ... through ...
pub const MOVE_BEAT_UP: MoveId = 251;

// === Effect ID Constants ===
pub const EFFECT_NORMAL_HIT: u8 = 0;
pub const EFFECT_BURN_HIT: u8 = 1;
pub const EFFECT_FREEZE_HIT: u8 = 2;
// ... all ~90 effect constants ...

// === Move Table (indexed by move_id - 1) ===
static MOVE_TABLE: [MoveData; NUM_MOVES] = [
    // #001 Pound
    MoveData {
        id: 1, name: "POUND",
        move_type: PokemonType::Normal, power: 40, accuracy: 100, pp: 35,
        is_special: false,
        effect_id: EFFECT_NORMAL_HIT, effect_chance: 0,
        priority: 1, high_crit: false,
    },
    // #002 Karate Chop
    MoveData {
        id: 2, name: "KARATE CHOP",
        move_type: PokemonType::Fighting, power: 50, accuracy: 100, pp: 25,
        is_special: false,
        effect_id: EFFECT_NORMAL_HIT, effect_chance: 0,
        priority: 1, high_crit: true,  // in CriticalHitMoves list
    },
    // ... #003 through #251 ...
];

pub fn move_data(id: MoveId) -> &'static MoveData {
    let idx = id as usize;
    if idx >= 1 && idx <= NUM_MOVES {
        &MOVE_TABLE[idx - 1]
    } else {
        &MOVE_TABLE[32] // fallback to Tackle (index 32 = move #33)
    }
}
```

## Priority Mapping

From `effects_priorities.asm`:

| Effect | Priority |
|--------|----------|
| EFFECT_PROTECT | 3 |
| EFFECT_ENDURE | 3 |
| EFFECT_PRIORITY_HIT | 2 |
| EFFECT_FORCE_SWITCH | 0 |
| EFFECT_COUNTER | 0 |
| EFFECT_MIRROR_COAT | 0 |
| All others | 1 |

The generator script will set `priority` based on the move's effect:
- Quick Attack, Mach Punch, ExtremeSpeed use EFFECT_PRIORITY_HIT → priority 2
- Protect, Detect use EFFECT_PROTECT → priority 3
- Endure uses EFFECT_ENDURE → priority 3
- Whirlwind, Roar use EFFECT_FORCE_SWITCH → priority 0
- Counter uses EFFECT_COUNTER → priority 0
- Mirror Coat uses EFFECT_MIRROR_COAT → priority 0
- Everything else → priority 1

## All 251 Moves — Yes, Not Just 1-125

**Rationale:** The data is all in a single file (`moves.asm`), already fully parsed by the generator script. Doing only 125 would require:
- An artificial cutoff in the script
- A fallback path for moves 126-251
- A second sprint to finish what's trivially completable now

Since the species database already references move IDs up to 251 in learnsets, having all move data available immediately is strictly better. There is zero additional complexity — same script, same array, same pattern. Just 126 more entries.

## Generator Script Approach

A Python script (`/tmp/gen_moves.py`) will:

1. **Parse `moves.asm`** — read each `move NAME, EFFECT, power, TYPE, accuracy, pp, effect_chance` line
2. **Parse `effects_priorities.asm`** — build priority lookup table
3. **Parse `critical_hit_moves.asm`** — build high-crit set
4. **Build type mapping:** ASM type names → Rust `PokemonType::*` variants
   - `NORMAL` → `Normal`, `FIRE` → `Fire`, etc.
   - `PSYCHIC_TYPE` → `Psychic` (ASM uses `PSYCHIC_TYPE` to avoid keyword conflict)
   - `CURSE_TYPE` → `Ghost` (special case)
5. **Derive `is_special`** from type (Gen 2 split)
6. **Derive `priority`** from effect (using priority lookup table, default 1)
7. **Derive `high_crit`** from move name (using critical hit set)
8. **Assign `effect_id`** — sequential u8 for each unique effect name
9. **Build move name** — convert ASM name to display name: `FIRE_PUNCH` → `"FIRE PUNCH"`, underscore to space
10. **Output** — write complete `move_data.rs` file

The script output will be directly pasted into the codebase. No runtime code generation.

1. **Update `MoveData` struct** in `data.rs` — add 4 new fields
2. **Update existing MoveData statics** in `data.rs` — add the 4 new fields to all 21 existing statics (temporary, will be removed in step 6)
3. **Run `cargo check`** — verify the struct change compiles
4. **Write and run `/tmp/gen_moves.py`** — parse ASM, generate `move_data.rs`
5. **Create `move_data.rs`** from script output
6. **Add `pub mod move_data;`** to `mod.rs`
7. **Update `data.rs`** — replace inline move data with re-exports from `move_data.rs`, remove old statics and `move_data()` function, remove old `MOVE_*` constants
8. **Update `species_data.rs` imports** — change `MOVE_*` imports to come from `move_data` instead of `data`
9. **Run `cargo test`** — all existing + new tests must pass
10. **Run `cargo build --target wasm32-unknown-unknown`** — verify WASM compiles

## Risks & Mitigations

**Risk:** Some move names in ASM don't match expected display names.
**Mitigation:** The script auto-converts `UNDERSCORES_TO_SPACES`. Manual exceptions: `PSYCHIC_M` → `"PSYCHIC"` (the ASM uses `_M` suffix to avoid keyword clash).

**Risk:** Existing code hardcodes `MOVE_TACKLE = 33` and other constants with specific values.
**Mitigation:** The new constants in `move_data.rs` will have identical values (derived from ASM position). Re-exports ensure no import path changes. The values 1-251 match the ASM ordering exactly.

**Risk:** `species_data.rs` imports `MOVE_*` constants from `data` — circular dep?
**Mitigation:** Import chain is `move_data.rs` imports types from `data.rs`, `species_data.rs` imports from `data.rs`, `data.rs` re-exports from both. This is the same pattern that already works for species. No circularity because `move_data.rs` and `species_data.rs` are siblings that both import from `data.rs`, and `data.rs` re-exports from both.

**Risk:** Adding 4 fields to MoveData breaks `battle.rs` pattern matching.
**Mitigation:** `battle.rs` accesses MoveData fields individually (`md.power`, `md.is_special`, etc.), never destructures the whole struct. New fields are simply available but unused until a future sprint implements effects.
