Engine Architecture

Complete technical reference for the Crusty Engine: ECS, rendering, physics, systems, and runtime architecture.

Overview

Crusty Engine is a Rust-native game engine that compiles to both WebAssembly (for browsers) and native targets (for headless testing and CLI). It features a custom ECS, software framebuffer renderer, fixed-timestep physics, and a 5-phase system execution pipeline.

The engine runs at 480×720 portrait orientation. All math uses f64. The framebuffer is a flat Vec<u8> in RGBA format, shared directly with JavaScript via WASM linear memory (zero-copy).

System Execution Pipeline

Every engine.tick(dt) executes 5 phases in order:

Phase 0: Input

  • Debug mode toggle (KeyD)
  • GestureRecognizer::update — advance gesture timers
  • Drain recognized gestures and publish to EventBus

Phase 1: Simulation (variable dt)

  • lifecycle — spawns, despawns, lifetimes, timers, behavior rules
  • hierarchy — parent-to-child transform propagation
  • signal — wire emitters to receivers
  • state_machine — tick elapsed time, check transitions
  • coroutine — advance async behavior steps
  • EnvironmentClock::tick — day/night cycle, seasons
  • FlowNetwork::solve — transfer resources along edges
  • sprite_animator — advance sprite frame timers
  • behavior — AI movement (chase, flee, patrol, etc.)
  • tween — easing-curve property animation
  • flash — hit flash, blink, color pulse
  • waypoint — path-following movement

Phase 2: Physics (fixed dt @ 60Hz)

Runs zero or more times per tick based on accumulator:

  • force_accumulator — sum external forces
  • integrator — semi-implicit Euler integration
  • collision — broadphase + narrowphase, emit events
  • physics_joint — distance, spring, rope, hinge constraints

Phase 3: PostPhysics

  • gameplay — collision reactions, scoring, damage
  • event_processor — legacy event triggers
  • input_gameplay — input-driven actions
  • Spawners — wave spawner, fire cooldown, player shooting
  • ghost_trail — position snapshots for trails
  • ParticlePool::update — tick particle lifetimes
  • TransitionManager::update, DialogueQueue::tick
  • Game-over check, Camera::update

Phase 4: RenderingPrep

Rendering order:

  • Framebuffer::clear — fill with background color
  • Starfield::render — parallax star background
  • renderer::run_entities_only — draw all visible entities
  • ParticlePool::render — particles
  • debug_render — debug overlays (if debug_mode)
  • HUD — bars, score, ammo, dialogue, game-over
  • ScreenFxStack::apply — tint, desaturate, flash
  • TransitionManager::apply — scene transitions
  • post_fx::apply — CRT scanlines, shake, bloom

Cleanup: clear events, end input frame, advance time/frame counters

Rendering Pipeline

Pure software renderer. No GPU. The Framebuffer is a Vec<u8> of RGBA pixels (width × height × 4 bytes). In WASM mode, JavaScript reads the pixel data directly from linear memory via framebuffer_ptr()/framebuffer_len() and blits to a canvas.

Physics

Fixed timestep at 60Hz (FIXED_DT = 1/60). Semi-implicit Euler integration. All math in f64.

  • Vec2 — 2D vector math (f64 only)
  • force_accumulator — sums gravity, force fields, impulses
  • integrator — updates position from velocity + forces
  • collision — AABB/circle broadphase + narrowphase, emits collision events
  • physics_joint — distance, spring, rope, and hinge constraints
  • ccd — continuous collision detection for fast-moving objects
  • spatial_grid — spatial partitioning for efficient queries

Subsystems

Game State (global_state)

A HashMap<String, StateValue> where StateValue is either Number(f64) or Text(String). The primary interface between game logic and headless testing.

engine.global_state.set_f64("score", 42.0);
engine.global_state.set_str("phase", "fight");
let score = engine.global_state.get_f64("score"); // Some(42.0)

Input

Tracks keyboard (key_down/key_up), mouse (position, buttons, delta), and touch events. The GestureRecognizer translates raw touch events into tap, long-press, swipe, and pinch gestures.

Sound

Command-buffer pattern. Rust queues SoundCommands (Play, Stop, SetVolume). JavaScript polls via drain_sound_commands() each frame and routes to Web Audio API. No direct JS callbacks from Rust.

Tilemap

Grid-based level representation. Each cell has a tile type, collision flag, and optional metadata. Used by S-League for both the overworld and fight course layouts.

UI Canvas

Declarative widget overlay system. Widgets are positioned relative to screen anchors and rendered on top of the game world. Supports buttons, labels, bars, and custom draw callbacks.

Key Design Decisions

  1. Single cdylib+rlib crate — same code compiles to WASM and native, no duplication
  2. Zero-copy framebuffer — JS reads pixel data directly from WASM linear memory
  3. All f64 math — no f32 anywhere, enforced by convention
  4. Sound as JSON drain — no JS callbacks, Rust queues commands, JS polls
  5. Fixed 60Hz physics — deterministic, independent of variable-dt simulation
  6. Game as Rust module — sleague.rs is a first-class engine module, not data
  7. Headless-first testing — all game logic testable via cargo test
